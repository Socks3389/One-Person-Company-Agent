# OPC OS × OpenSpec 集成方案

## 为什么集成 OpenSpec？

| OPC OS 的痛点 | OpenSpec 的解决方案 |
|---------------|---------------------|
| PRD 和架构文档写完就放着了，没人跟踪 | 转化为可执行的 `tasks.md`，逐项打勾实现 |
| 只有大的阶段划分，没有细粒度变更管理 | 每个变更独立文件夹，完整生命周期管理 |
| 没有标准化的规格格式，全靠自由发挥 | proposal → specs → design → tasks 标准化结构 |
| 做完就忘，没有历史归档 | 自动归档到 `archive/YYYY-MM-DD-xxx/` |

```
用户输入 /opc 做一个博客系统
    │
    ▼
┌─────────────────────────────────────────────────┐

                          │
                     v0-ceo-brief.md
                     v1-prd.md
                     v1-arch.md
                          │
                          ▼
┌─────────────────────────────────────────────────┐
│          OpenSpec 集成层 (规格驱动)               │
│  ─────────────────────────────────────────────  │
│  openspec init                                  │
│  /opsx:propose blog-v1                          │
└─────────────────────────┬───────────────────────┘
                          │
                ┌─────────┴─────────┐
                ▼                   ▼
        openspec/changes/blog-v1/
            ├── proposal.md    ← 来自 CEO 简报
            ├── specs/         ← 来自 PRD
            ├── design.md      ← 来自架构设计
            └── tasks.md       ← 自动拆解的可执行任务
                          │
                          ▼
              用户输入 /opsx:apply
                          │
                          ▼
              按 tasks.md 逐项实现
                          │
                          ▼
              完成后 /opsx:archive
                          │
                          ▼
              归档到 archive/2025-xx-xx-blog-v1/
                          │
                          ▼
              OPC OS 更新各角色私有记忆
              (经验沉淀、教训总结)
```

## 映射关系：OPC OS → OpenSpec

| OPC OS 交付物 | → | OpenSpec 工件 | 说明 |
|---------------|---|---------------|------|
| v0-ceo-brief.md | → | `proposal.md` | 动机、目标、业务价值 |
| v1-prd.md | → | `specs/*.md` | 用户故事、场景、验收条件 |
| v1-arch.md | → | `design.md` | 技术选型、模块划分、API 契约 |
| 架构 + PRD | → | `tasks.md` | 自动拆解为 checkbox 任务清单 |

## 完整工作流示例

### Step 1: OPC OS 启动项目

```
用户: /opc 做一个个人博客系统

Claude: 🚀 One Person Company OS 启动
        问 3 个问题...
        ✅ CEO 商业决策完成
        ✅ PRD 已生成
        ✅ 架构设计已生成
        
        💡 正在生成 OpenSpec 规格...
```

### Step 2: 自动生成 OpenSpec

自动执行：
```bash
openspec init
/opsx:propose blog-v1
```

自动写入文件：
```
openspec/changes/blog-v1/
├── proposal.md    # 为什么做这个，要解决什么问题
├── specs/
│   ├── user-stories.md   # 来自 PRD 的用户故事
│   └── acceptance.md     # 验收条件
├── design.md      # 技术选型、模块划分、数据模型
└── tasks.md       # 1. 初始化项目
                   # 2. 实现文章 CRUD
                   # 3. 实现用户认证
                   # 4. 实现 UI
                   # ...
```

### Step 3: 按规格实现

```
用户: /opsx:apply

Claude: 按 tasks.md 逐项实现...
        ✅ 1. 初始化项目
        ✅ 2. 实现文章 CRUD
        ...
```

### Step 4: 验证与归档

```
用户: 完成了

Claude: ✅ 运行 /opsx:verify 验证
        ✅ 验证通过
        ✅ 运行 /opsx:archive 归档
        ✅ 已归档到 openspec/archive/2025-xx-xx-blog-v1/
        ✅ 已更新各角色私有记忆（经验沉淀）
        🎉 项目完成！
```

## 关键设计点

### 1. 无缝切换，用户无感知

- 用户不需要知道内部是 OPC OS 还是 OpenSpec
- 只要输入 `/opc ...`，剩下的自动化
- 到了实现阶段自动提示 `/opsx:apply`

### 2. 双向同步

- OPC OS 的输出自动填充 OpenSpec 的输入
- OpenSpec 的完成状态自动同步回 OPC OS 的项目状态

### 3. 双重记忆

- **OPC OS 记忆**：角色私有记忆（跨项目的经验沉淀）
- **OpenSpec 归档**：每个变更的完整历史（本项目的可追溯）

## 为什么这是黄金组合？

| 维度 | 只有 OPC OS | 只有 OpenSpec | OPC OS + OpenSpec |
|------|-------------|---------------|-------------------|
| **流程管控** | ✅ 7阶段完整流程 | ❌ 只有规格层 | ✅ 完整端到端 |
| **角色视角** | ✅ 角色隔离+独立记忆 | ❌ 单视角 | ✅ 每个角色产出专业规格 |
| **规格驱动** | ❌ 自由格式 | ✅ 标准化工件 | ✅ 标准化+可执行 |
| **任务拆解** | ❌ 只到阶段层面 | ✅ 细粒度任务清单 | ✅ 从战略到战术全覆盖 |
| **经验积累** | ✅ 跨项目记忆 | ❌ 只有本项目归档 | ✅ 双重历史沉淀 |
| **质量门禁** | ✅ 80分卡壳机制 | ❌ 没有质量校验 | ✅ 流程+规格双重保障 |
| **使用体验** | ✅ 简单，一条命令 | ✅ 简单，斜杠命令 | ✅ 无缝集成，自动切换 |

---

> **底层逻辑**：OPC OS 负责"做什么"和"为什么做"的战略层面，OpenSpec 负责"怎么做"和"做到哪了"的战术层面。两者结合，形成从创意到发布的完整闭环！
